Job system

Architecture

Five sections over one Bun.serve. The chat WebSocket and the dataset LLM templates both ride the server's own loopback /v1, so everything serializes through the single generation queue — one GPU, no second path.

src/web/app.html ── unified SPA (hash routes #/chat #/quantize #/finetune #/dataset #/status) │ ▼ src/server.ts ── Bun.serve({ fetch, websocket }) ├─ /ws/chat .............. src/pi-web.ts → embedded pi AgentSession → loopback /v1 ├─ /api/dataset/* ........ src/dataset/* → jobs (in-process) ├─ /api/quantize/* ....... src/quantize/* → jobs (GPU subprocess) ├─ /api/finetune/* ....... src/train/* → jobs (GPU subprocess) ├─ /api/jobs/:id[/stream] .. src/jobs/* → SQLite rows + NDJSON-log SSE └─ /v1/* · /stats · /fit · /library (unchanged) │ ▼ src/mlx/{ffi,ops,autograd}.ts → mlx-c via bun:ffi mlx_quantize · mlx_save_safetensors · mlx_value_and_grad

The headline: native, no Python

mlx-bun's thesis is that the buggy layer (loading, tokenizing, the training loop, serving) doesn't need Python. The two hardest pieces — weight quantization and gradient training — are now native, proven on a real model.

Build journal

How it was built, the decisions, and the bugs found along the way.

  • 🔭

    Plan first, fork the big decision to the user

    Surfaced the one decision that changed everything — native compute vs. orchestrating the installed Python vs. proxying — and you chose native, no Python. Full pi agent for chat; real tiny runs for verification.

  • 🔬

    Gated the riskiest unknown before building

    Native training needs autograd through bun:ffi. A feasibility spike proved mlx_value_and_grad + a JSCallback closure returns correct grads (finite-diff < 1e-2) before a line of trainer code — de-risking the whole subsystem.

  • ⚡

    Parallel sub-agents on non-overlapping files

    After laying a shared job contract (src/jobs/types.ts), fanned out six subsystems concurrently — jobs, dataset, quantizer, trainer, web SPA, pi embed — each owning its own directory, then integrated server.ts/cli.ts centrally to avoid conflicts.

  • 🧱

    Found: mlx_save_safetensors exists

    The quantizer needed a safetensors writer; reading the mlx-c header (open book) showed a native writer already exists — no hand-rolled byte layout, eliminating a whole class of dtype/alignment bugs.

  • 🛡️

    Crash isolation by design

    Async Metal OOM is uncatchable (it std::terminates the process). So GPU jobs run as child mlx-bun subprocesses with a kernel fault boundary; a job that OOMs dies alone and the server survives. Verified with a deliberate-crash test.

  • 🐛

    Bug: LoRA init key shape

    The first real training run failed — random_normal got a [1,2] key but mlx wants [2]. Reshaped the per-module subkey after slicing the split-keys array. Loss curve immediately worked.

  • 🐛

    Bug: flaky chat test

    The pi agent sometimes takes a tool-call turn before producing text; the test resolved on the first turn_end and saw no tokens. Fixed to settle on the first turn that actually streams text — the embed was always correct.

  • 🔁

    Test, fix, re-run

    Each real-model e2e surfaced a small integration gap (a path not resolved, a key shape) — fixed at the source and re-run green, rather than worked around in the test.
