1Executive summary

mlx-bun is already architected for distribution — the hard problems are solved in code. The "one binary" decision (2026-06-12) means the executable ships alone (~61 MB, 100% project code + Bun); the MLX native runtime (libmlx, libmlxc, libjaccl, mlx.metallib — ~52 MB compressed) is a separate versioned tarball that the binary downloads itself on first run, resumable and sha256-verified through the same machinery as model downloads (src/native-pack.ts). And mlx-bun serve with no arguments auto-downloads the RAM-appropriate recommended model (recommendedRepoId(): e4b < 24 GB, 12B at 24 GB, 26B at 48 GB+) and serves it. The end-user experience is already install one small binary → run one command → everything else self-provisions.

Distribution is therefore mostly publishing plumbing, not engineering: ship that 61 MB binary through (1) a public GitHub repository with tagged releases as the source of truth — including the native-v<ver> release tags the first-run downloader already points at, (2) an npm package for the TypeScript library + bunx CLI audience, (3) a Homebrew tap whose formula installs just the binary, and (4) the existing build-binary.sh full bundle (binary + dylibs side-by-side, nothing downloaded) as the embeddable sidecar artifact for Mac app developers. Everything hinges on three prerequisites: making the repo public (the native-pack URL works anonymously only then), automating builds in CI, and acquiring an Apple Developer ID for signing/notarization. A Mac App Store app and Docker images are explicitly out of scope (Section 9).

2Distribution constraints unique to this project

These shape every option below; any plan that ignores one of them fails in the field.

Self-provisioning native runtime

The runtime needs libmlxc.dylib → libmlx.dylib → libjaccl.dylib + mlx.metallib — and the project already solves this: ensureNativeRuntime() downloads a versioned, sha256-pinned native pack (~52 MB tar.gz) on first run into ~/Library/Caches/mlx-bun/native-v<ver>-<arch>/. Resolution order: MLX_BUN_LIBMLXC env → beside the executable (embedders) → native-pack cache → Homebrew (dev trees). End users never touch brew. The distribution-relevant consequences: every release must publish a matching native-v<ver> asset, and the baked-in sha256 makes version skew structurally impossible for pack users (brew-resolved dev trees remain the skew risk).

Bun version floor

Requires Bun ≥ 1.3.14 (Bun.Image, verified FFI). Source/npm installs must enforce this (engines field + a runtime check); compiled binaries embed their Bun, sidestepping it entirely.

Models self-bootstrap too

Weights are 7–18 GB and live in the user's HF cache; distribution ships the resumable, checksummed downloader, never the weights. Better: mlx-bun serve with no arguments already auto-downloads the recommended model for the machine's RAM (recommendedRepoId() — e4b below 24 GB, 12B at 24 GB, 26B MoE at 48 GB+), scans, fit-checks, and serves it. The advertised first-run UX for every channel is one command: install → mlx-bun serve. get/scan/ls remain the explicit-control path.

macOS Gatekeeper

Any binary downloaded outside brew/npm gets quarantined. Unsigned binaries show the "cannot be opened" dialog. Distribution-grade binaries need a Developer ID certificate, codesigning of the executable and every dylib, and notarization. (The build script currently ad-hoc signs — fine locally, not for distribution.)

Apple Silicon only, on purpose

No Linux, no Windows, no Intel Macs, no containers. This is a feature (scope as survival strategy) but must be loudly stated in every channel's metadata (os/cpu in package.json, depends_on arch: :arm64 in brew) so users fail at install time, not runtime.

Pre-alpha honesty

Versioning starts at 0.x with explicit instability messaging. Bit-exact logit parity is the project's credibility asset — release gates should run the parity suite, and the README's measured-numbers culture should carry into release notes.

4Option A — GitHub repository + tagged source releases Foundation

Everything else (npm provenance, brew formula URLs, release assets, issue tracking) hangs off this. The repo is in good shape content-wise — README quickstart, PLAN.md, benchmark provenance, THIRD_PARTY_LICENSES.md all exist.

Step-by-step

  1. Pre-publication audit. Sweep history and tree for anything private: git log -p | grep -iE 'token|sk-|key', check for machine-specific absolute paths in committed files, and prune working artifacts that don't belong in a public tree (the one-off plan-status-*.html, wwdc-*.html, duplicate benchmark files — keep the canonical dated ones referenced by the README). Confirm fixtures/ and goldens/ contain nothing license-encumbered (they're derived from model outputs — note their provenance in a short goldens/README).
  2. Add the community scaffolding. CONTRIBUTING.md (state the parity-oracle rule: every model-code PR must pass bun test goldens; perf claims need numbers), a minimal SECURITY.md, issue templates (bug report must ask for bun --version, brew info mlx-c, machine + RAM), and a CODE_OF_CONDUCT.md.
  3. Decide the repo home and publish. Personal account vs. an org (an org, e.g. mlx-bun, ages better if collaborators join and lets the npm scope and brew tap share the name). gh repo create mlx-bun/mlx-bun --public --source . --push. Fill in the git clone <this repo> placeholder in the README.
  4. Branch protection + CI gate. Protect main; require the test workflow (Section 10) to pass. macOS arm64 runners (macos-14/macos-15) run the fast test tier; model-loaded tests auto-skip without the snapshot, which is exactly right for CI.
  5. Publish the native pack release the code already points at. src/native-pack.ts hardcodes releases/download/native-v0.1.0/mlx-bun-native-v0.1.0-arm64.tar.gz with a baked sha256/size. Run ./scripts/build-native-pack.sh 0.1.0, create the native-v0.1.0 release with that asset, and verify the constants match. While the repo is private, first-run downloads need GITHUB_TOKEN / gh auth token (the code handles the presigned-URL dance); going public is what makes anonymous first-run work — test it from a fresh user account immediately after flipping visibility.
  6. Tag v0.1.0 and cut the first release. Use gh release create v0.1.0 --notes-file RELEASE_NOTES.md. Release notes follow the README's culture: what's verified bit-exact, measured numbers with machine provenance, known-open items (e4b decode residual, Qwen pending). Adopt Keep a Changelog in a CHANGELOG.md from day one.

Exit criterion: a stranger with an M-series Mac can go from the repo URL to a served model following only the README, and files an issue when they can't.

5Option B — npm package Phase 1

Serves the library audience (the differentiator — no other npm package does native MLX inference) and gives a no-clone CLI via bunx mlx-bun serve …. Published as TypeScript source: Bun executes TS natively, so no build step — but this makes the package Bun-only, which must be explicit.

Step-by-step

  1. Reserve the name. Check npm view mlx-bun; if taken, fall back to a scope (@mlx-bun/core). Create the npm org matching the GitHub org and enable 2FA + granular automation tokens.
  2. Restructure package.json for publication. Remove "private": true; add: 
    {
  "version": "0.1.0",
  "exports": {
    ".": "./src/index.ts",
    "./server": "./src/server.ts"
  },
  "bin": { "mlx-bun": "./src/cli.ts" },
  "files": ["src", "docs", "README.md", "LICENSE", "THIRD_PARTY_LICENSES.md"],
  "engines": { "bun": ">=1.3.14" },
  "os": ["darwin"], "cpu": ["arm64"],
  "repository": "github:mlx-bun/mlx-bun",
  "keywords": ["mlx", "llm", "apple-silicon", "bun", "local-inference", "gemma"]
}
    Exclude tests/, goldens/, fixtures/, spikes/, benchmark files via the files allowlist — the published tarball should be a few hundred KB, not the repo.
  3. Create a public entrypoint. Add src/index.ts re-exporting the supported surface (Gemma4Model, Weights, generate, loadTokenizer, ChatTemplate, loadModelConfig, KV store). What's exported here is the semver contract; everything else stays internal. Update README's library section from "clone and import from src/" to bun add mlx-bun.
  4. Guard the one real failure mode: Node instead of Bun. The native runtime is not a failure mode — the CLI paths already call ensureNativeRuntime() before touching FFI. But ensure the library entrypoint does too (or documents calling it), since library consumers bypass the CLI's ensureNative step. Add a preflight: if typeof Bun === "undefined", throw with "mlx-bun requires Bun ≥ 1.3.14 (https://bun.sh)" instead of a cryptic bun:ffi resolution error under Node.
  5. Dry-run and publish. npm pack --dry-run to inspect the tarball; install the pack into a scratch project and run the README library example end-to-end. Then npm publish --access public --provenance (provenance requires publishing from the GitHub Actions workflow with id-token: write — preferred; see Section 10).
  6. Verify the two consumption modes. On a clean machine (or fresh user account): bunx mlx-bun serve should go from nothing to a serving model with zero other commands (native pack download → recommended-model download → serve), and bun add mlx-bun + library import should work. Document both in the README under a new "Install" section that leads with the one-command path; the clone path stays for contributors.
Versioning policy: 0.x semver — minor bumps may break the library API, patches never do. State this in the README. Move to 1.0 only when the exported surface in src/index.ts has survived a model-family addition (Qwen) without breaking — that's the proof the abstraction is right.

6Option C — Homebrew tap Phase 2

The best-fit channel for the server audience, and thanks to the self-provisioning runtime, a trivially simple formula: it installs only the 61 MB compiled binary from the Option D release. No mlx-c dependency, no Bun dependency, no vendored dylibs — the binary downloads its own native pack on first run. Start with a personal tap — no approval process, ships in a day; homebrew-core can come later once the project has the notability core requires.

One judgment call: brew purists prefer packages that don't phone home post-install. The first-run download is well-precedented (toolchains, model runners do this), but disclose it in caveats; an alternative is a formula resource block that installs the native pack into the keg at install time — more brew-idiomatic, at the cost of duplicating what the binary already does. Recommend the simple binary-only formula first.

Step-by-step

  1. Create the tap repo: github.com/mlx-bun/homebrew-tap with a Formula/ directory. Users then do brew install mlx-bun/tap/mlx-bun.
  2. Write the formula against the Option D release artifact: 
    class MlxBun < Formula
  desc "Native MLX inference server for Apple Silicon (OpenAI/Anthropic-compatible)"
  homepage "https://github.com/mlx-bun/mlx-bun"
  url "https://github.com/joshuarossi/mlx-bun/releases/download/v0.1.0/mlx-bun-v0.1.0-darwin-arm64.tar.gz"
  sha256 "<sha256 of the tarball>"
  license "MIT"
  depends_on arch: :arm64
  depends_on macos: :sonoma

  def install
    bin.install "mlx-bun"     # single binary; native runtime + models
  end                         # self-download on first run

  def caveats
    <<~EOS
      First run downloads the MLX native runtime (~52 MB, sha256-verified)
      to ~/Library/Caches/mlx-bun and, if no model is present, the
      recommended model for this Mac's RAM (7-18 GB, resumable).
      Just run:  mlx-bun serve
    EOS
  end

  test do
    assert_match version.to_s, shell_out("#{bin}/mlx-bun --version")
  end
end
  3. Test locally: brew install --build-from-source Formula/mlx-bun.rb, then brew test mlx-bun and brew audit --strict.
  4. Automate the bump. Add a job to the release workflow that, after assets upload, opens a PR against the tap updating url + sha256 (a 20-line script or the bump-formula-pr pattern). Releases then propagate to brew with zero manual steps.
  5. (Later) homebrew-core. Once the repo has real-world traction, submit to core for plain brew install mlx-bun. Core disallows vendored prebuilt binaries in some cases — at that point revisit a source-build formula with a bun build dependency. Not a Phase 2 concern.

7Option D — Standalone signed binary releases Phase 2

Per the one-binary architecture, a release is two independent artifacts, both already producible by existing scripts:

ArtifactBuilt bySizeCadenceConsumed by
mlx-bun-v<ver>-darwin-arm64.tar.gz — the compiled CLI alonebun build --compile (first stage of build-binary.sh)~61 MBEvery release tagbrew tap, curl-install, direct download
mlx-bun-native-v<ver>-arm64.tar.gz — the MLX runtime packscripts/build-native-pack.sh~52 MB compressedOnly when the mlx/mlx-c version bumps (own native-v* tags; sha256 + size baked into src/native-pack.ts)The binary itself, on first run
dist/ full sidecar bundle — binary + dylibs + metallib side-by-sidescripts/build-binary.sh~230 MBEvery release tag (or on request)Embedders (Option E) — they never download at runtime

The builds work today; the gap is signing, notarization, and CI automation. Both the executable and the native pack's dylibs need real signatures (currently ad-hoc): the binary because users download it through a browser/brew and Gatekeeper checks it, the dylibs because a hardened-runtime binary with library validation refuses ad-hoc-signed libraries.

Step-by-step

  1. Enroll in the Apple Developer Program ($99/yr) and create a Developer ID Application certificate. Export it as a .p12; store the .p12 (base64), its password, your Apple ID, team ID, and an app-specific password as GitHub Actions secrets.
  2. Add a signing stage shared by both build scripts (a sibling sign-bundle.sh used by build-binary.sh and build-native-pack.sh): replace the ad-hoc codesign -s - with the real identity, sign inside-out — each dylib first, then the executable — with hardened runtime: 
    IDENTITY="Developer ID Application: Josh Rossi (TEAMID)"
for lib in libjaccl.dylib libmlx.dylib libmlxc.dylib; do
  codesign --force --options runtime --timestamp -s "$IDENTITY" "$OUT/$lib"
done
codesign --force --options runtime --timestamp \
  --entitlements entitlements.plist -s "$IDENTITY" "$OUT/mlx-bun"
    Known trap: bun build --compile binaries are self-modifying-adjacent (embedded bytecode); verify the signed binary still launches, and add com.apple.security.cs.allow-jit (JavaScriptCore needs JIT) and, if FFI requires it, allow-unsigned-executable-memory / disable-library-validation to entitlements.plist. Test each entitlement empirically — start minimal, add only what an actual launch failure demands.
  3. Notarize: zip the bundle, submit, staple is N/A for bare executables (stapling applies to .app/.dmg/.pkg — Gatekeeper checks notarization online for tarball'd binaries): 
    ditto -c -k --keepParent dist mlx-bun.zip
xcrun notarytool submit mlx-bun.zip --apple-id "$APPLE_ID" \
  --team-id "$TEAM_ID" --password "$APP_PW" --wait
    Then verify on a quarantined copy: xattr -w com.apple.quarantine ... on a downloaded copy and confirm it launches without the Gatekeeper dialog (spctl -a -vv -t install on the components).
  4. CI release workflow (.github/workflows/release.yml): on tag push v* — macOS arm64 runner → bun install → fast test tier → brew install mlx-c → build bundle → import signing cert into a temp keychain → sign → notarize → tar.gz with the version in the name + a SHA256SUMS file → gh release upload. The artifact name encodes platform: mlx-bun-v0.1.0-darwin-arm64.tar.gz.
  5. Native-pack version discipline. The pack's sha256/size are baked into src/native-pack.ts, so pack users can never skew — but the release process must keep them in lockstep: when mlx/mlx-c bumps, run build-native-pack.sh <newver>, publish the native-v<newver> release, bake the printed constants, then tag the binary release. Record brew info mlx-c provenance in the native-pack release notes. The remaining skew surface is dev trees resolving from /opt/homebrew/lib — keep the startup version assertion for that path.
  6. Optional curl-install script: a short install.sh on the repo (curl -fsSL https://raw.githubusercontent.com/.../install.sh | sh) that detects arm64/macOS, downloads the latest release, verifies the SHA256, extracts to ~/.mlx-bun/bin, and prints the PATH line. Nice-to-have once brew exists; ship it only if users ask.
  7. Smoke-test the release artifact itself as a CI job: download the just-uploaded tarball on a fresh runner, run mlx-bun --version and mlx-bun fit --help (no model needed), assert exit 0. A full serve smoke test needs weights — keep a tiny fixture model or skip serve in CI and gate it on the manual pre-release checklist instead.
Gatekeeper nuance worth knowing: files fetched by the binary itself (the native pack, models) never receive the com.apple.quarantine attribute — only browser/curl-downloaded files do. So Gatekeeper assessment applies to the binary, while the pack's dylibs are gated by the binary's hardened-runtime library-validation policy instead. Sign both properly and neither path bites; test the matrix explicitly (browser-downloaded binary + self-downloaded pack) before declaring Phase 2 done.

8Option E — Embeddable SDK for Mac apps Phase 3

This is the strategically differentiated channel: a signed, notarization-ready inference sidecar that Tauri/Electron developers drop into Resources/ and spawn. docs/embedding.md already documents the pattern (spawn, poll /v1/models, --memory-budget as the OOM defense, SIGKILL-safe restarts). The work is productizing it.

Step-by-step

  1. Make the Option D artifact the embedding artifact. Same bundle; document that embedders re-sign the components with their Developer ID as part of their app's signing pass (standard for bundled helpers), and which entitlements their host app needs to inherit (the JIT entitlement from Section 7).
  2. Publish a tiny client/launcher npm package (@mlx-bun/sidecar): spawn with port selection, readiness polling, graceful + hard shutdown, and typed re-exports of nothing else — the host app talks OpenAI/Anthropic protocol with its existing SDK. ~200 lines, huge DX win, keeps the big binary out of npm.
  3. Ship a reference example app in examples/ (one Tauri, one Electron, each < 100 lines): bundle the sidecar, first-run model download with progress (the downloader's resumability shines here), chat UI via the OpenAI SDK pointed at localhost. Examples are the real documentation for this audience.
  4. Document the sandbox boundary honestly: a sandboxed Mac App Store host cannot use the shared HF cache path without entitlements/security-scoped bookmarks; the supported configuration is non-MAS (Developer ID) distribution, or the host passes an explicit container-relative model directory (HF_HOME override). Put this in docs/embedding.md before someone hits it in the wild.
  5. Stability contract: declare the HTTP API (chat completions, messages, responses, /stats, adapters) as the embedding-stable surface, versioned independently of the library API — embedders are insulated from internal refactors by the protocol.

10Cross-cutting work

CI/CD pipeline (one workflow, all channels)

# .github/workflows/release.yml — trigger: push tag v*
jobs:
  test:      # macos-15 arm64 — bun install, brew install mlx-c, bun test (fast tier)
  build:     # needs: test — build-binary.sh, sign, notarize, tar + SHA256SUMS
  release:   # gh release create with CHANGELOG excerpt + assets
  npm:       # npm publish --provenance (id-token: write)
  brew-bump: # PR to mlx-bun/homebrew-tap with new url + sha256

Plus a plain PR workflow running the fast test tier on every push. Model-loaded parity tests stay local (the snapshot is 8.3 GB) — codify a manual pre-release checklist in RELEASING.md: full bun test with the oracle snapshot present, a served smoke request against each protocol, and a glance at ./benchmark.sh results if perf-relevant code changed.

Versioning & release cadence

  • Single version across CLI, library, server, and binary bundle — one tag drives all channels.
  • 0.x semver as defined in Section 5; CHANGELOG.md per Keep-a-Changelog; release notes carry measured numbers with machine provenance, in keeping with the project's culture.
  • Cadence: tag when a PLAN.md phase completes, not on a calendar.

Model availability dependency

The product depends on the mlx-community OptiQ repos remaining available. Mitigations: mlx-bun get already checksum-verifies; record the expected snapshot hashes for the three supported models in-repo; document the gated-repo auth path. If an upstream repo is ever yanked, mirroring becomes a Day-0 incident — acceptable risk now, note it in the register.

Documentation per channel

Rewrite the README's install section as a channel picker: brew (server users) / npm (library users) / release tarball (embedders, no-toolchain) / clone (contributors). Each path ends at the same single command — mlx-bun serve — with the native runtime and recommended model self-provisioning on first run (get/scan/fit documented as the explicit-control path). The current README quickstart (brew install mlx-c → clone → get → scan → serve) becomes the contributor path only. docs/embedding.md gains the signing/entitlements material from Section 8.
