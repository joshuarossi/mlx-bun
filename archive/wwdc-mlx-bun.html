The problem

Apple Silicon is extraordinary.
JavaScript can’t reach it.

MLX is Apple’s ML framework — hand-tuned Metal kernels for the unified-memory GPU. It ships official bindings for four languages. The language of every agent, every dev tool, every Electron and Tauri app isn’t one of them.

✓ Python ✓ C++ ✓ Swift ✓ C ✗ JavaScript

So today, a TypeScript app that wants local inference shells out to a Python server — and inherits that stack’s fragility: venv setup, brittle downloads, monkey-patched HTTP layers, segfaults on exit.

The insight

The GPU sets the speed.
Delete everything around it.

Every matmul, every attention pass already lives in MLX’s C++ and Metal core, exposed through mlx-c. The Python on top is pure orchestration — loading, tokenizing, sampling, serving. The GPU dominates; that layer is performance-neutral.

Which means it can be rewritten in any runtime without losing a single token per second. So we rewrote it in the one your code already lives in.

Decode is memory-bandwidth-bound: ~273 GB/s on an M4 Pro ÷ bytes-of-weights = the ceiling. Wins come from fewer bytes per token — never from “faster JS.”

Findings

What the port taught us.

01
Bun.mmap traps past 4 GB

Weight shards run 5.35 GB. We bind libc mmap through bun:ffi and pass MLX raw pointers — JS views only for small ranges.

02
Metal reads garbage off unaligned pointers

Silently. Weights load via MLX’s lazy loader instead; our own KV files are page-aligned, so cache reloads are true zero-copy GPU mmaps.

03
The JIT deleted our reads

After tier-up, JSC elides typed-array reads across FFI calls. Every out-param readback now uses read.u64. Filed upstream as bun#32054.

04
bf16 rounds x·x·x ≠ x³

One op-order slip compounds across 48 layers into wrong logits. The rule: port the implementation, never the docstring.

05
Wired memory is load-bearing

The 26B decoded at 8.6 tok/s — Metal re-faulting weights every token — until one wired-limit call. Then 32.3. Same code.

06
GPU OOM is uncatchable

MLX throws from Metal’s completion thread; no handler is on that stack. Admission control before generation is the only defense.

07
Speculative decoding can lose

0.51–0.91× on e4b — the drafter can’t outrun a 54 tok/s target. Measured against an identical reference trace, recorded, kept behind a flag.

08
Our 4% decode “gap” was an accounting story

An allocator stall mlx-lm bills to prompt time, we billed to decode. Fixed reference-faithfully — the 12B now decodes at-or-above the reference, paired.

Every finding lives in PLAN.md with its root cause, fix, and the test that pins it. The losing experiments stay too — as documented, default-off flags. Measurement over memory.

The road ahead

What’s next.

Qwen 3.x
The second model family — and the home of MTP multi-token prediction: its quants ship a trained drafting head, so speculative decoding finally gets a drafter fast enough to win.
SigLIP vision
Eyes for e4b and the 26B — extending the verified, token-exact vision path beyond the 12B’s sidecar.
Compiled decode graphs
mlx_compile moves per-step graph construction into C++ — erasing the last ~5% of host overhead on small, fast models.
npm
The library-first API, published. Today it’s clone-and-import; tomorrow it’s bun add mlx-bun.
More silicon
The benchmark matrix extends across machines, and fit --skus already predicts memory and throughput for the entire Apple Silicon lineup — before you download a byte.
