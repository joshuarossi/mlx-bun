01 the headline
+2.3% fused kernel over compat · paired @8k cleared-machine confirmation · MLX_BUN_PERF_KERNEL earns default-on

Small number, big rule.

The project's standing rule is that defaults change on clean-machine numbers only. The Phase E flash-decoding kernel — one Metal dispatch per quantized SDPA, dequant inlined, deliberately rebuilt after v1 lost on occupancy — showed 1.02–1.04× in paired dirty-machine A/Bs. Today's cleared pass confirms it: 1.023× @8k, 1.010× @2k, paired against compat in the same process, quality gate green throughout.

That closes the loop the optimization plan opened: compile captured the host-side cost (Phases A–D), and the fused kernel captured what was left on the GPU side. Both levers are now measured, confirmed, and defaulted — and the experiments that lost stay in the tree as documented flags, not deletions.

07 reading this honestly

What the numbers don't say

  • One machine this time. This is the M4 Pro file only — no M1 Max cross-check ran today. One machine per file; cross-machine claims go through the per-row machine column.
  • The kernel win is ~1–2%, not 2–3×. The capture measured why: mlx's quantized_matmul already fuses dequant, so the fat round-trip the kernel was built to remove never existed in our port. The real prize was score/softmax round-trips and dispatch count — about what it delivered.
  • The kernel path is not bit-exact — bit-exactness is unreachable at bf16 rounding ties. It's gated by teacher-forced agreement against the frozen compat oracle (60–62/64), at the same envelope as the already-shipped tiled path (62–63/64). Compat mode remains the bit-exact reference.
  • Don't read the 26B prefill column as a stack comparison. Both Python stacks measured 19 tok/s prefill on the 26B this pass (mlx-lm read 193 yesterday on the same machine) — the Python-side MoE prefill measurement is unstable. mlx-bun's own 26B prefill held at 208–220 across both passes.
  • mlx-lm still wins 12B direct decode (25.8 vs 25.4, −1.6%) and e4b short-prompt prefill (384 vs 290). optiq edges the 12B server by 0.3 tok/s. The sweep is broad, not total.
  • mlx-lm's 26B server holds 5.5 GB RSS to our 18.3 GB. mlx-bun keeps weights resident; the Python server pages more lazily. That's a deliberate latency trade — but it is a trade, and on a 24 GB machine it matters.
  • Logit parity remains the correctness oracle. Default-path outputs are verified against the mlx-lm reference; that's what makes a head-to-head fair at all.
