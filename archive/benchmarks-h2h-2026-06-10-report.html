mlx-bun runs quantized LLMs on Apple Silicon straight from TypeScript — no Python interpreter, no sidecar process, one Bun runtime. The question this benchmark answers: does dropping Python cost you anything? Decode speed lands at parity. Everything around the generation loop — cold start, time-to-first-token, server overhead — gets dramatically faster.

01 the headline
4.8× faster to first token server leg · vs the Python stacks

The win isn't decode. It's everything else.

On the HTTP server leg, mlx-bun answers with first-token latency of 57–88 ms. mlx-lm and optiq — both Python — sit at 281–376 ms. Same models, same machine, same prompts. No interpreter to spin up, no tokenizer round-trip through a subprocess, no framework tax between the socket and the Metal kernel.

Cold start tells the same story: mlx-bun is ready in ~0.5 s; the Python servers take 1.3–5.6 s. And once it's warm, decode throughput is a dead heat — within a couple of percent either way.

On raw in-process decode, mlx-lm's hand-tuned Python kernels edge ahead by 2–12% (most on the small e4b model, where per-token overhead bites hardest). But on the server leg — the configuration anyone actually deploys — mlx-bun flips it: +9% on the 26B MoE (50.2 vs 45.9 tok/s) and a tie on e4b, because the Python stacks lose on the wire what they won in the kernel.

What holds — and what diverges

  • The TTFT win is machine-independent. mlx-bun returns the first token in 45–89 ms on both parts; the Python stacks need 220–376 ms on both. The ~4–5× gap is structural — it doesn't depend on the silicon.
  • Decode parity holds on both. mlx-bun lands within ~2–12% of mlx-lm's direct decode on every model/machine, and retakes the lead on the server leg — true on the M1 Max and the M4 Pro.
  • 8k prefill is 1.78× on both — to the decimal. mlx-bun ingests an 8k prompt at 253 tok/s vs mlx-lm's 143 on the M4 Pro, and 186 vs 107 on the M1 Max. Different absolute throughput, identical ratio.
  • Bandwidth shows up where you'd expect — and where you wouldn't. The 12B decodes ~14% faster on the higher-bandwidth M1 Max (29.2 vs 25.6 tok/s). But the 26B mixture-of-experts is faster on the 24 GB M4 Pro (55.1 vs 50.5) — its active-expert compute, not memory bandwidth, sets the ceiling.
  • The 26B prefill regression is M1-Max-specific. mlx-bun actually beats mlx-lm on the M4 Pro (231 vs 186 tok/s) but collapses to 49 on the M1 Max. Suspected wired-memory pressure from 17.7 GB of weights on the 32 GB part — not a routing defect. Under investigation, not buried.
  • Robustness diverges in mlx-bun's favor. On the M4 Pro, optiq's server crashed loading the 26B with a Metal out-of-memory abort; mlx-bun served the same model from the same machine state at 55.1 tok/s via lazy load and a scoped wired-memory limit.

08 reading this honestly

What the numbers don't say

  • This is an M1 Max, 32 GB — not the M4 Pro the project README quotes. One machine per file; cross-machine numbers go through the machine column. Don't blend them.
  • mlx-lm wins raw direct decode. Its Python kernels are mature and fast. mlx-bun's edge is the end-to-end serving path, not the inner loop.
  • 26B prefill cratered on this machine — 49 tok/s against mlx-lm's 119. But the cross-check (§07) shows mlx-bun winning the same metric on the M4 Pro (231 vs 186), so this reads as M1-Max memory pressure, not a routing defect. Flagged, scoped, not buried.
  • Python servers hold less resident memory on the 26B (8.1–11.2 GB vs mlx-bun's 17.2 GB). mlx-bun keeps more weights warm in RSS; that's a deliberate trade for latency, but it is a trade.
  • Logit parity is the correctness oracle. Speed means nothing if the outputs drift — mlx-bun's logits are bit-exact against the mlx-lm reference, which is what makes a head-to-head fair at all.
