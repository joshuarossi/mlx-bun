pi as a built-in terminal for mlx-bun

Can mlx-bun pi drop the user into a working pi agent session against the locally running model — and what's the best mechanism? Sources: pi.dev/docs/latest (sdk · tui · rpc · custom-provider) and direct inspection of the installed @earendil-works/* packages, v0.79.1.

TL;DR — yes, and better than hoped. pi is a Bun-native, MIT-licensed npm package whose own release binary is built with bun build --compile — the exact toolchain mlx-bun's single-binary bundle already uses. The full interactive TUI (InteractiveMode) is an exported SDK class, not CLI internals. Four integration options exist, they compose, and the plan is: ship the cheap paths now (harness helper + subprocess launcher), promote the SDK embed to the single-binary flagship. The locally installed pi is the v1 stepping stone, never the destination — though users with their own pi stay first-class forever. pi's RPC/event modes separately give us the backbone for the built-in web chat UI.

What pi is, technically

  • Packagespi-coding-agent 13 MB · pi-ai 4.9 MB · pi-agent-core 1.3 MB · pi-tui 1.9 MB — ~21 MB total, all MIT
  • Bun-compile nativeUpstream's build:binary is literally bun build --compile — embedding is a supported configuration, not an experiment
  • RuntimeNode ≥ 22.19 required (fails on 18: /v regex); fine under Bun ≥ 1.3.x, which mlx-bun pins
  • Local model proven~/.pi/agent/models.json on this machine already runs an mlx provider (openai-completions) against local Qwen quants
  • Binary assetsTheme JSONs, PNGs, export-html templates, photon wasm, pi-tui darwin-arm64/x64 .node prebuilds — upstream's copy-binary-assets script is the manifest
  • Headless modes-p print mode, --mode json, --mode rpc; SDK exports runPrintMode / runRpcMode

The four integration options

1 · mlx-bun harness pi — configure the user's existing pi do regardless

Drop a tiny extension into ~/.pi/agent/extensions/ that calls pi.registerProvider("mlx-bun", …) with dynamic discovery from our /v1/models — no models.json surgery, reversible by deleting one file. Validate with a tiny request, print the launch hint.

Pros

  • Nearly free (hours of work)
  • Respects the user's pi sessions, themes, extensions
  • Zero version coupling
  • Model list auto-syncs with the server

Cons

  • Requires pi to already be installed
  • Not "built-in" — two moving parts
  • Config-merge bugs are the classic footgun (extension route mostly avoids it)

2 · mlx-bun pi v1 — spawn the user's installed pi right v1

Ensure the server, spawn pi with a session-scoped provider (generated -e extension, no global writes) and inherited stdio. Unrecognized flags pass straight through — so mlx-bun pi -p "…", --mode rpc, --continue, @file args all work on day one.

Pros

  • ~A day to ship; immediate demo value
  • Full pi feature surface via passthrough
  • pi upgrades arrive for free
  • Simple failure modes (no pi → install hint)

Cons

  • Still two installs
  • No control over the user's pi version — upstream changes can break the launcher
  • Per-run provider injection needs care
  • Server + child lifecycle/shutdown handling

3 · SDK embed — pi inside the mlx-bun binary flagship

createAgentSession() in full-control mode (in-memory registry/settings, custom ResourceLoader, own session dir — nothing leaks from/to ~/.pi unless opted in), then run new InteractiveMode(runtime) — pi's own TUI, in-process, against the in-process server over loopback. -p and --mode rpc/json stay supported natively via runPrintMode/runRpcMode. Open questions are tractable: TUI latency while the same JS thread builds decode graphs (test, don't redesign), bundle plumbing for pi's assets (upstream's binary build is the existence proof), version pinning, and session-dir identity.

Pros

  • The true appliance: one binary, no prior installs
  • We pin and test the exact pi version we ship
  • Full control of defaults per device profile
  • In-process events shared with the web UI
  • Offline-clean

Cons

  • Most engineering (~1–2 weeks incl. bundling + latency gates)
  • We own the upgrade treadmill of a fast-moving 0.x dep
  • Diverges from the user's own pi config unless opted in
  • Binary grows ~20 MB

4 · RPC / JSON event stream web-UI backbone

JSONL over stdin/stdout: prompt, steer, abort, model control, stats; streamed message_update / tool_execution_* / turn_* events. Wrong tool for the terminal (we'd rebuild pi's TUI on pi), but the right backbone for the built-in web chat UI — bridge WebSocket ↔ events, or drive AgentSession.subscribe() in-process with the same event shapes.

Pros

  • Documented, versioned protocol
  • UI-agnostic: web, IDE, bots
  • Steering/abort/stats built in
  • Works over subprocess or embedded session

Cons

  • Not a terminal UX
  • Strict JSONL framing rules
  • Extension-UI sub-protocol must be bridged

Implementation plan

To be explicit: the locally installed pi is the v1 stepping stone, not the destination. P1 support never goes away; the flagship ends embedded.

P1

mlx-bun harness pi · hours–1 day

  1. Detect pi (PATH, ~/.bun/bin/pi, npm global); report version.
  2. Generate a dynamic-discovery extension (registerProvider + fetch /v1/models at pi startup). Reversible: delete one file.
  3. Validate with a tiny completion through pi's provider path; print the launch hint.
  4. Failure modes: pi missing → install one-liner; server down → offer to start it.
P2

mlx-bun pi v1 — subprocess launcher · ~1–2 days

  1. Ensure server: reuse a healthy running instance, else start in-process and wait.
  2. Resolve the pi binary; offer the installer if absent.
  3. Spawn with inherited stdio + session-scoped provider via generated -e extension (no global writes), --provider mlx-bun --model <profile default>.
  4. Pass through all remaining argv — -p, --mode rpc, --continue, @file work for free.
  5. Lifecycle: SIGINT to the child; optional keep-warm flag; clean server shutdown.
P3

Embed spike · ≤1 week, via bun run, no bundling yet

  1. Pin pi packages exact (0.79.x).
  2. createAgentSession full-control: in-memory registry/settings, loopback provider, session dir under ~/.mlx-bun/, our system prompt.
  3. Run InteractiveMode in the same process as the server.
  4. Gates (numbers on this machine): editor latency clean during 12B decode; decode tok/s within noise of server-only; tool round-trip works; clean teardown.
  5. Decide discovery policy: isolated by default, --pi-config opt-in to the user's ~/.pi.
P4

Single binary + shared event plumbing · ~1 week

  1. Fold pi's asset manifest into the bun build --compile bundle; verify asset-path resolution inside the binary.
  2. CLI: mlx-bun pi → embedded InteractiveMode; -p → runPrintMode; --mode rpc → runRpcMode; P2 path kept as --external-pi.
  3. Web chat UI rides the same AgentSession.subscribe() events over WebSocket — no subprocess.
  4. Upgrade policy: bump pi deliberately; the P3 gates become the regression suite.

First-run starter model folds into P2/P4 · model gated on Phase 14

First run must not be "watch a progress bar for 10 minutes." A sub-gigabyte starter model downloads first and is chatting within ~1 minute — docs-grounded tour-guide prompt, banner: "the ideal model for your Mac is downloading in the background." Downloads are sequenced, not parallel; swap when the profile model lands. Starter is downloaded, never bundled (a ~900 MB binary kills brew install) and stays on disk afterward (future speculative-decoding draft + always-works fallback).

Does it make sense as a feature?

Yes — arguably the highest-leverage product move available: it turns mlx-bun from "a server you point things at" into the one-binary local-AI appliance the roadmap describes. The hard parts are already done on both sides (our OpenAI surface is SDK-verified; pi's SDK is designed for embedding; same Bun toolchain; MIT license). Caveats: we inherit pi's release cadence; the showcase quality ceiling is the local model's tool-calling (the Lucien-direction evals would quantify it); the embedded TUI must not degrade decode throughput — measured, per house rules.
