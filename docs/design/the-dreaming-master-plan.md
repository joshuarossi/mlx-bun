## PREAMBLE

### Goal
Build a local "personal Wikipedia" (codename **the Dreaming**) inside `mlx-bun` (`src/memory/`) that synthesizes a user's (Josh's) chat history into **entity articles** and lets a local chat agent answer questions by **small, focused reads** — never a context dump. The cloud original is **Lucien** (`/Users/joshrossi/Code/lucien`, live articles in `~/Dreaming`, ground-truth DB `/Users/joshrossi/Code/lucien/.lucien/lucien.db`). The bar: read one article and get up to speed on a thing, the way Wikipedia works.

### Settled principles (FINAL — do not re-derive or contradict)
- **Articles are THINGS (nouns/entities):** `Panasonic_Lumix_S5IIX`, `L-Mount`, `Sigma_150-600`, `Photography`. Every article = "<the thing> AND how the user relates to it." Write the RELATIONSHIP (why chosen, how used, opinions, plans); LINK OUT to real Wikipedia for general world-facts.
- **Notability is personal** (the ONE deviation from Wikipedia): a thing is article-worthy because the user engaged with it — owned / used / decided-on / recurring (≥2 conversations) / holds a substantive opinion. This is the only deviation; otherwise **do whatever Wikipedia does**.
- **NO VECTOR SEARCH on the routing/read hot path.** "About lenses" = a CATEGORY / TYPE / TITLE lookup (deterministic, zero embeddings). Embeddings are at most an OFFLINE TEST INSTRUMENT, never the router. The recall experiment's job was to PROVE topic-bins are wrong (~67% top-12 recall on broad bins — an *indicative* prior-session figure with no committed source in `reports/`/`scripts/`; it is RE-MEASURED live in P4-T2, so nothing gates on the literal), which it did.
- **Capacity by decomposition:** a small local model's capacity scales inversely with task size. Decompose every stage to the smallest single-purpose task, ideally BINARY ("does this chunk update this section? yes/no"), run M×N. Affordable because local inference is ~free. Coarse gates first (hierarchical pruning: "does this relate to this article?" gates the section checks).
- **Small focused units, never a dump — read AND write.** The read path hands the model one short focused thing per hop (one article, one section, one infobox). The write path emits small focused units. No giant blobs in either direction.
- **Self-healing through ordinary use.** The conversation IS the input — the user's in-conversation CORRECTIONS are normal chunks like any other. When a new chunk CONTRADICTS an existing article claim, synthesis updates the article TOWARD the user's latest/corrected position (Bayesian-style — resolve toward the most recent statement; not append-both, not ignore). So the wiki heals itself with no manual editing. Example: the AI reads an article and says *"you love 35mm for photography"*; the user replies *"it's fine but nowhere near as good as 50mm"*; that correction is an ordinary chunk that folds in via the nightly synthesis; the next day the AI reads the updated article and says *"how great is that 50mm."*
- **AI does JUDGMENT; deterministic CODE enforces invariants** (footnote bijection, wikilink canonicalization, infobox-traces-to-cited-claim, ≥70% word floor, weak output → NO-OP). "LoRAs propose, gates dispose."
- **Entities (do not conflate):** **chunk** (a conversation segment) → **bucket** (optional staging group, may ship empty) → **article** (a THING) → **section**. Many-to-many at every layer. bucket ≠ article.
  - **L4 stated deviation (intentional, not drift):** this plan demotes *bucket* from Lucien's "a group of chunks about one topic" to "OPTIONAL staging only, may ship empty." Articles, not buckets, are the synthesis target here. A future reader should treat this redefinition as a deliberate design choice recorded in this doc, not an inconsistency to repair.
- **Vault = truth, DB = derived index.** Markdown in the vault is canonical and git-tracked; every DB table is either *pipeline state* or a *derived cache* rebuildable from the markdown by a `reindex` pass.

### The entity data model
- **chunk**: a pointer range `(conv, start, end)` + a 4–10-word `label`. No text copy.
- **entity**: a canonical name (== article H1 == filename stem with `_`→space), with aliases and a `kind ∈ {thing, person, domain, project, standard}`.
- **article**: one markdown file per entity; fixed skeleton (H1 → series banner → infobox → lead → sections → See also → Notes → References).
- **infobox**: typed key-value facts that the AI READS as part of the article's content (like a Wikipedia infobox), never a query/filter engine. **world-facts** (`type`, `mount`, `focal_length`, …) trace to citable external facts; **relationship-facts** (`owned`, `acquired`, `used_for`, `chosen_over`, `opinion`, …) are the personal-notability layer and must trace to a cited body claim. A value naming another entity is a `[[wikilink]]` whose only structural role is to feed the link graph for NAVIGATION — there is no numeric-facet query over infobox fields.
- **category**: a Wikipedia-style grouping of things, **emergent** (created by judgment as articles are written, governed by the editable `Category_Definitions.md`; the wiki starts with zero). Membership is **explicit** — the article declares `[[Category:Name]]`. "About lenses" = `Category:Lenses` membership (deterministic lookup over whatever categories exist).

### North-star acceptance test (design backward from this)
User asks the chat AI: *"what's the best lens for really long reach that isn't crazy expensive?"* The local agent SILENTLY (like a colleague — no "per the wiki…") FINDS the one relevant article — by category ("articles about lenses/telephoto"), name, or a `[[link]]` from Josh's camera article — then READS it (its prose AND its infobox, as content) and answers from what it read. The article already holds the user's CURRENT POSITION on the topic — the synthesized conclusion he reached before (e.g. "you already shoot the 75-300, so for real reach the Sigma 150-600 is the sane pick") — so the answer is just reporting back what the article says. It is **find → read → answer** in SMALL reads: NOT a faceted filter that computes "Sigma 150-600" by ordering lenses on a price field, NOT a 90-article substring dump, NOT a 671-line blob.

**Where the work is.** The read path is DELIBERATELY TRIVIAL — find the right article (category / name / link), read it, answer — exactly like reading one Wikipedia page. All the complexity, machinery, and acceptance pressure live on the WRITE side: synthesis keeping each article a good, current, well-shaped account of the user's relationship to the thing. The thesis is proved by the WRITE→READ round-trip: fold a new chunk (e.g. "I bought the 70-300") → it patches the affected article(s) → the same simple read now returns the evolved answer.

### Output-parsing contracts (H1 — bake once, used by every model stage)
In `src/memory/parse.ts`:
- `parseBinary(out: string): boolean` — `return /^\s*y/i.test(out);` (anything not starting with y/Y → `false` → NO-OP). Binary stages call `generateText(tm, prompt, {maxTokens: 4})`.
- `parseLines(out: string): string[]` — split on `\n`, `trim`, drop empties and any line equal (casefold) to `NONE`. List stages call `generateText(tm, prompt, {maxTokens: 128})`.
- A malformed/over-long generation is truncated by `maxTokens`; `parseBinary` defaults it to `false`, `parseLines` drops it. No stage ever blocks on bad output.

### Lucien source schema (B9 — inlined; verified via `sqlite3 .lucien/lucien.db .schema`)
The bootstrap reads these tables READ-ONLY. **Note the column names differ from mlx-bun's** (mapping baked into P5-T1):
```
conversations(uuid PK, name, summary, created_at TEXT, updated_at TEXT, message_count INT, source TEXT)
messages(uuid PK, conversation_uuid, position INT, sender, text, timestamp TEXT, parent_message_uuid)   -- INDEX(conversation_uuid, position)
chunks(id INTEGER PK AUTOINCREMENT, conversation_uuid, start_message_uuid, end_message_uuid, label)       -- chunks point at MESSAGE UUIDs, not positions
buckets(name PK, description)
chunk_buckets(chunk_id, bucket_name, PK(chunk_id,bucket_name))    -- the "~8809 edges"; reinterpreted as entity edges in P4-T2
chunked_conversations(conversation_uuid PK, chunked_at TEXT, status)
synthesized_bucket_chunks(bucket_name, chunk_id, synthesized_at, PK(bucket_name,chunk_id))
```
There is **no `edges` table and no `entities` table** in Lucien — entity gold is MINED from `~/Dreaming` markdown (P0-T5), and "edges" means `chunk_buckets`. mlx-bun `chunks` use integer `(start,end)` POSITIONS; Lucien `chunks` use `(start_message_uuid,end_message_uuid)` — P5-T1 maps UUID→position via `messages(conversation_uuid, position)`.

**`(chunk, section_before, section_after)` triple-recovery rule (B9 — explicit, used by P7-T2 / P10-T1).** Recover the chunk's identity and its article from Lucien's STRUCTURED tables, NOT by parsing footnote hashes — a `conv:HASH` carries only the conversation (see the citation format in P1-T1), never a chunk range, so it cannot identify one chunk of the many in a conversation. Source of record (verified via `sqlite3 /Users/joshrossi/Code/lucien/.lucien/lucien.db .schema`):
- **(chunk → article):** `chunks(id, conversation_uuid, start_message_uuid, end_message_uuid, label)` ⋈ `chunk_buckets(chunk_id, bucket_name)` ⋈ the bucket→article-stem mapping (the bucket that became `articles/<stem>.md`); `synthesized_bucket_chunks(bucket_name, chunk_id, synthesized_at)` is the ledger of which chunks were actually integrated, and when.
- **(chunk text/span):** the chunk's `start_message_uuid`/`end_message_uuid` resolved over `messages` (the P5-T1 UUID→position map) — never copied from the article prose.
- **(section_before, section_after):** from the article's GIT HISTORY. Run in a clone/worktree of `~/Dreaming` at its frozen SHA; for each commit in `git log --reverse` touching `articles/<stem>.md` (or `<stem>.md`) that ADDED a `conv:` footnote-def (the presence of an added `[^N]:` footnote merely flags an *integration* commit vs a pure normalize/rename commit, which is skipped), the diff hunk's enclosing `##`/`###` heading is the target **section**, `section_before` = that section's body at the commit's parent (`git show <parent>:<path>` → `extractSection`), `section_after` = its body at the commit.
- **(pairing):** align the article's integration commits (in `git log` order) with its `synthesized_bucket_chunks` rows for that bucket (ordered by `synthesized_at`) — each integration commit ↔ one integrated chunk row. The footnote only marks *that* a commit integrated a chunk; the chunk's IDENTITY and span come from the structured `chunks` row, never from the hash.
Each surviving `(chunk, section_before, section_after)` triple is one SFT example.

### Training-data schema (B4 — exact, verified against `src/train/dataset.ts sftRowFormat` + Lucien `train.jsonl`)
The trainer (`finetuneRunner`) reads `${DATA}/train.jsonl` + `${DATA}/valid.jsonl`, one JSON object per line, format auto-probed by `probeFormat`. Accepted shapes: `{messages:[…]}`, `{prompt,completion}`, or `{text}`. **Use the `messages` shape** (matches the existing chunk dataset):
```json
{"messages":[{"role":"system","content":"<stage system prompt + inlined Meta policy>"},{"role":"user","content":"<stage input>"},{"role":"assistant","content":"<gold output>"}]}
```
Loss is masked to the assistant turn (`promptLen`). Per-stage field mapping:
- **synthesis (P10-T1):** user = current section markdown + the integrating chunk text (`chunkText`) + footnote contract; assistant = the gold `section_after` (the integrated section + new `[^N]:` lines), from the triple.
- **section (P10-T2):** user = article title + section heading + section gist + chunk label/gist; assistant = `yes`/`no` (one token) from the gold `chunk_sections` edge.
- **entity (P10-T3):** user = chunk text; assistant = newline-listed gold entity names for that chunk.
- **editor (P10-T4):** user = section markdown; assistant = the tightened/infobox-extracted gold output.
