## Why it's different

<div class="mlx-features">
  <div class="mlx-feature">
    <p class="lbl">Runtime</p>
    <h3>No Python</h3>
    <p>Bun + <code>mlx-c</code> over <code>bun:ffi</code>. No venv, no sidecar Python server, no segfaults on exit. One binary, one toolchain.</p>
  </div>
  <div class="mlx-feature">
    <p class="lbl">Correctness</p>
    <h3>Bit-exact</h3>
    <p>Matched bit-for-bit to the reference it's run against — mlx-lm in bf16, mlx-optiq when quantized. Identical logits, not "approximately."</p>
  </div>
  <div class="mlx-feature">
    <p class="lbl">Speed</p>
    <h3>Fastest to first token</h3>
    <p>Served over HTTP, the fastest TTFT and startup of any stack tested — 45–90 ms vs Python's 220–330 ms — with near-zero server tax.</p>
  </div>
  <div class="mlx-feature">
    <p class="lbl">Protocols</p>
    <h3>OpenAI + Anthropic</h3>
    <p>Drop-in chat completions, Anthropic Messages, and OpenAI Responses. Point any SDK — or Claude Code — at a fully local backend.</p>
  </div>
  <div class="mlx-feature">
    <p class="lbl">Multimodal</p>
    <h3>Tools &amp; vision</h3>
    <p>Native tool-calling parsed into OpenAI <code>tool_calls</code>; image input via native OS codecs (HEIC, AVIF, WebP…) on vision models.</p>
  </div>
  <div class="mlx-feature">
    <p class="lbl">Shape</p>
    <h3>Library-first</h3>
    <p>The server is one consumer of a TypeScript API. Import generation into a Bun process, or embed the binary as a Mac-app sidecar.</p>
  </div>
</div>

## Measured, not vibes

Head-to-head on an M4 Pro (24 GB), same models, same day, against mlx-lm and mlx-optiq:

| | mlx-bun | mlx-lm | optiq |
|---|---|---|---|
| **TTFT, served (warm)** | **45–90 ms** | 219–224 ms | 222–331 ms |
| **server start → ready** | **0.36–0.47 s** | 0.76–0.98 s | 0.79–1.00 s |
| **server tax vs direct decode** | **≈ 0%** | −5…−7% | ≈ 0% |

The honest negatives are in the same table — see [Benchmarks](/about/benchmarks/).
